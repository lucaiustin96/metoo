1. Idei/Concepte importante privind design-ul general

MeToo este o aplicatie menita a incuraja interactiunea dintre persoanele care studiaza/lucreaza in localuri comune disponibile intr-un areal geografic. Odata ajuns intr-un astfel de spatiu, fiecare utilizator va putea sa anunte public ce activitatea desfasoara. Toti utilizatorii interesati de aceste activitati, in functie de abilitatile pe care le detin, vor primi notificari si se vor putea alatura utilizatorului care a postat anuntul. Totodata, aplicatia va contine o sectiune in care vor fi afisate cele mai populare astfel de locuri.

2. Maniera de interactiune cu utilizatorul

Aplicatia va contine urmatoarele pagini:

  • Home
  • Activities
  • Edit profile
  • Top Locations
1. Home este pagina generala a aplicatiei, aceasta continand descrierea aplicatiei.
2. Activities este pagina pe care va fi redirectionat utilizatorul dupa ce s-a conectat. Aceasta va contine un buton de adaugare locatie, un buton de adaugare anunt, un select din care userul isi alege locatia in care se afla, iar in partea dreapta contine o sectiune in care sunt afisate postarile altor utilizatori care se afla la locatia specificata de persoana logata. La click, butonul de adaugare locatie va deschide un formular in care utilizatorul va putea adauga o locatie care nu exista. De asemenea, butonul de adaugare anunt va deschide o fereastra in care utilizatorul va putea sa adauge activitatea pe care o desfasoara in acel moment. Aplicatia contine si niste iconite pe care utilizatorul poate apasa in cazul in care vrea sa contacteze un alt utilizator care a postat un anunt.
3. Edit profile este o pagina a aplicatiei care ii permite utilizatorului sa modifice datele personale, si anume, first name, last name si sa poata adauga abilitati pe care acesta le detine.
4. Top locations este pagina care va contine top patru locatii populare din oras, reprezentative pentru astfel de activitati desfasurate.
5. Aplicatia va contine si doua ferestre care se vor deschide cand un utilizator doreste sa se autentifice sau sa se inregistreze. Ele vor putea fi accesate din meniu.

Aplicatia este gandita pente a oferi utilizatorului senzatia ca foloseste o aplicatie pentru telefonul mobil. Toate paginile sunt responsive atat pe mobil cat si pe tableta. Ferestrele care se deschid atunci cand utilizatorul trimite sau primeste un mesaj sau doreste sa adauge un anunt sau o locatie, sunt gandite pentru a nu forta utilizatorul sa faca miscari in plus pentru a le accesa.

5. Descrierea diagramelor si a bazei de date

Diagrama de deployment:
In nodul cu baza de date se gasesc tabelele cu userii, locatiile si postarile utilizatorilor iar la artifact se regaseste codul care se ocupa de logica paginii(Server Logic). Pe langa nodul care contine baza de date, mai exista un nod care apartine serverului de la Facebook. El contine un artifact in care se afla Facebook API.
Diagrame de clase:
Pentru fiecare functionalitate a site-ului exista cate o clasa (Login, Sign up, AddPlace, AppPost, DisplayTopLocations) pe backend in care sa se proceseze datele inainte sa fie inserate in baza de date. In acest sens, se trimit formulare din js catre clasa php, iar de acolo se insereaza in baza de date.
Diagrama use case:
Utilizatorul are posibilitatea de a se loga si de a face sign up prin intermediul Facebook API. Apoi datele introduse in momentul editarii profilului, adaugarii postarilor, adaugarii locatiilor, vizualizarii postarilor, vizualizarii top locations, trimiterii de mesaje, se salveaza in baza de date. Mesajele si notificarile sunt transmise userului 2 de catre userul 1.
Baza de date
Utilizatorul va fi identificat printr-un id unic care va fi obtinut de la Facebook API. Baza noastra de date va contine 2 tabele: tabelul in care vor fi salvate locatiile si tabelul in care vor fi salvate postarile. Tabelul locatii va contine un ID care este cheie primara, locatia salvata sub forma unui sir de caractere si va contine o coloana in care va fi salvat ID-ul utilizatorului care a introdus locatia. Tabelul postari va contine un ID unic care este cheie primara, continutul postarii salvat sub forma unui sir de caractere, ID-ul utilizatorului obtinut de la Facebook API si ID-ul locatiei postarii. Am ales sa nu salvam conversatiile in baza de date deoarece o baza de date SQL nu ar fi potrivita.

6. Detalii referitoare la modelarea datelor proprii

Modelarea datelor proprii
Locatiile si postarile sunt introduse de catre utilizator dupa ce acesta s-a logat. Datele sunt transmise prin intermediul unui formular catre o pagina PHP unde sunt prelucrate si securizate iar mai apoi sunt transmise catre baza de date. Pentru editarea profilului se foloseste tot un formular care actualizeaza sau adauga date in baza de date. Pagina ofera posibiliatea vizualizarii celor mai vizitate locatii (Top Locations). Pentru a realiza aceasta functionalitate, este nevoie doar de interogarea bazei de date si de returnarea rezultatelor. Aplicatia ofera posibilitatea initializarii unei conversatii intre 2 utilizatori. Conversatia va fi salvata atat timp cat utilizatorii vor fi logati. In momentul delogarii, conversatia se sterge.
Facebook API
Inainte de a folosi Facebook API, vom avea nevoie de un Facebook App ID. Pe pagina de la Facebook API trebuie sa introducem un URL catre care utiizatorul va fi redirectionat dupa ce se logheaza. In aplicatia noastra trebuie sa introducem fisierul de la Facebook SDK.JS. In paginile noastre trebuie sa verificam daca utilizatorul este logat sau nu. In cazul in care utilizatorul nu este logat, Facebook API va deschide o fereastra in care utilizatorul isi va putea introduce datele. De asemenea, Facebook API va contine un script care va oferi utilizatorului posibilitatea de a se deloga.
Scheme de modelare a microdatelor folosind HTML5
Deoarece majoritatea functionalitatilor oferite de aplicatia noastra nu sunt accesibile decat daca utilizatorul este logat, schemele de microdate nu sunt folosite foarte mult. Singura functionalitate care le foloseste este cea care afiseaza topul locatiilor. Pentru aceasta vom folosi itemtype ="http://schema.org/Place", iar fiecare locatie din lista va folosi proprietatea itemprop="name" pentru a ajuta la recunoasterea mai usoara a locatiilor de catre motoarele de cautare.

7. Solutii de stocare disponibile la nivel de client

Service Worker
Vom implementa un service worker in scopul utilizarii aplicatiei atat online, cat si offline. Acesta se va instala automat pe dispozitivul clientului la prima accesare a aplicatiei. Toate informatiile vor fi stocate in cache, astfel ca dupa ce utilizatorul a intrat cel putin o data in site, acesta va putea accesa aplicatia si fara a avea acces la Internet.
Web Storage
Web Storage ofera doua modalitati de stocare diferite: Local Storage si Session Storage. Acesta difera in functie de domeniu si de durata de viata. Session storage mentine stocarea datelor pe durata sesiunii de pagina, adica atat timp cat browserul este deschis, inclusiv reincarcarea si restabilirea paginilor. Local Storage face acelasi lucru pe care il face Session Storage cu diferenta ca stocarea datelor persista chiar si dupa ce browserul a fost inchis si redeschis. Vom utiliza Local Storage pentru a stoca paginile aplicatiei.
